# VARSYS — Бэклог

Известные проблемы и задачи по доводке (после первого подъёма на железе, v0.2.2).
Статус: устройство загружается и базово отзывается; впереди отладка UI/UX.

## Открытые задачи

### 1. В настройках наслаиваются шрифты
В экране настроек текст/строки накладываются друг на друга.
- Где смотреть: `src/ui/screens/SettingsScreen.cpp` (две прокручиваемые колонки,
  `makeRow`, высоты строк/скролл). Вероятно, проблема в layout строк при прокрутке.

### 2. Экран батареи показывает только «100%»
Не отображаются напряжение, ток и прочие параметры — только процент.
- Где смотреть: `src/modules/PowerModule/PowerModule.cpp` — чтение BQ27220
  (`bqRead16` для `BQ_VOLTAGE` / `BQ_CURRENT`, возможно неверные регистры/масштаб),
  `src/ui/screens/BatteryScreen.cpp` — вывод V/мА.

### 3. Нет настройки часов — всегда 12:00
Время не устанавливается, статус-бар постоянно показывает 12:00.
- Нужно: ручная установка времени и/или NTP при наличии WiFi, опц. сохранение в RTC.
- Где смотреть: `src/core/Clock.*`, статус-бар `src/ui/StatusOverlay*`.

### 4. Не продуман сон / длительный сон
Сейчас «сон» = только погашенный экран + CPU 80 МГц. Нет настоящего
`esp_light_sleep` / `deep_sleep`, нет пробуждения по GPIO (энкодер/кнопка),
нет сценария «глубокий сон» для долгого хранения.
- Где смотреть: `src/modules/PowerModule/PowerModule.cpp`.

### 5. Кнопка «назад» работает не везде
На части экранов возврат не срабатывает.
- Где смотреть: централизованная обработка `INPUT_BACK` → `popScreen` в
  `src/ui/UIManager.cpp`; экраны, открытые через `setScreen` (а не `pushScreen`),
  не попадают в стек навигации. Проверить переходы по всем разделам.

### 6. Непонятно, работают ли функции разделов
Нужно пройтись по КАЖДОМУ пункту меню на железе и проверить реальную
работу (разбирается отдельно, по одному разделу за раз).

### 7. Принудительный перезапуск при зависании
Аварийный выход/ребут — например, удержание кнопки «назад» 5–7 секунд → `ESP.restart()`.
- Нюанс: программный обработчик длинного нажатия спасёт только при «мягком»
  зависании (главный цикл ещё крутится, застряли на экране/в долгой операции).
- Для НАСТОЯЩЕГО зависания цикла нужен аппаратный сторож:
  - Task Watchdog (TWDT) на loop-задачу — авто-ребут, если цикл не «кормит» WDT.
  - Либо отдельная проверка удержания «назад» через прерывание/таймер, не зависящая
    от основного цикла.
- Где смотреть: `src/modules/InputModule/InputModule.cpp` (детект long-press back),
  `src/core/Core.cpp` (loop, инициализация WDT).

## Найдено превентивным аудитом кода (2026-06-28)

### 8. QWIIC 43/44: GPS и NRF24 не работают (конфликт пинов при init)
Пины 43/44 делят GPS(UART) / NRF24(CE,CSN) / iButton(OneWire). Все три модуля
конфигурируют пины в `init()` при загрузке → кто инициализируется ПОСЛЕДНИМ,
тот владеет пинами. Порядок gps→nrf→ibutton ⇒ пин 44 = OneWire (iButton),
а GPS-RX и NRF-CSN затёрты. По сути работает только iButton.
- Фикс: ленивая инициализация (как для WiFi/BLE-радио) — поднимать пины
  только при заходе в раздел (NrfScreen/GpsScreen/IButtonScreen onShow),
  отпускать на onHide. Тогда один QWIIC-модуль за раз.
- Где смотреть: `NrfModule::init` (s_radio.begin клобберит CE/CSN),
  `GpsModule::init` (UART), `IButtonModule::init` (OneWire), порядок в `Core.cpp`.

### 9. RMT-канал FastLED vs CC1101/IR — ИСПРАВЛЕНО в аудите
FastLED держит RMT_CHANNEL_0 (вспышка LED при загрузке) → CC1101 (был ch0)
не мог поставить драйвер → запись/воспроизведение Sub-GHz не работали бы.
Уведено: CC1101→ch2, IR→ch3, FastLED=ch0. Проверить запись Sub-GHz/IR на железе.

### 10. Wardrive морозит UI и не останавливается при выходе
Блокирующий `wifi.scan()` (~2с) в Scheduler-колбэке каждые 4с подвешивает
интерфейс; при выходе с экрана `WardriveScreen::onHide` не зовёт `deactivate()`,
скан продолжается в фоне. Вынести скан в задачу / останавливать на onHide.

### 11. Аудио-пины при будущем включении звука
GPIO40 = LCD_RST = SPK_WCLK (звук сейчас отключён из-за этого), плюс
GPIO39 = MIC_CLK = SPK_MCLK. Перед включением аудио нужны корректные I2S-пины
платы, иначе ломается дисплей. См. `AudioModule`.

### 12. Strapping-пины (низкий риск, иметь в виду)
- GPIO0 = нажатие энкодера: удержание при включении → download mode (норм, мы
  этим пользуемся для прошивки).
- GPIO45 = PN532 RST — это strap VDD_SPI; на практике PN532 определяется, но
  при проблемах с флешем/питанием смотреть сюда.
