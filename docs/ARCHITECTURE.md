# Архитектура VARSYS

Документ описывает принципы модульной прошивки VARSYS на этапе 1.

## Слои

```
┌─────────────────────────────────────────────┐
│                  Экраны UI                    │  ← BootScreen, HomeScreen ...
├─────────────────────────────────────────────┤
│         Модули (IModule): UIManager, ...      │  ← бизнес-логика подсистем
├─────────────────────────────────────────────┤
│      Ядро: Core / ModuleManager / EventBus    │  ← оркестрация и связь
├─────────────────────────────────────────────┤
│        HAL: Display, board_pins, ...          │  ← абстракция железа
├─────────────────────────────────────────────┤
│        Arduino-ESP32 / LovyanGFX / LVGL       │  ← платформа и библиотеки
└─────────────────────────────────────────────┘
```

## Ключевые сущности

### Core (`src/core/Core.*`)
Синглтон-оркестратор. В `begin()` инициализирует логгер, регистрирует модули,
вызывает `initAll()`/`startAll()`. В `loop()` с периодом `VARSYS_TICK_MS`
прокручивает `updateAll()`.

### IModule (`src/core/Module.h`)
Контракт любой подсистемы: `init() → start() → update() → stop()`.
Чтобы добавить новую функцию прошивки, достаточно:
1. создать класс-наследник `IModule`;
2. зарегистрировать его в `Core::registerModules()`.

### EventBus (`src/core/EventBus.*`)
Слабая связанность модулей. Источники событий (например, будущий модуль ввода)
публикуют `Event`, а подписчики (экраны UI) реагируют. Модули не держат
прямых ссылок друг на друга.

### ModuleManager (`src/core/ModuleManager.*`)
Хранит список модулей, управляет их жизненным циклом и предоставляет
`find(name)` для редких случаев прямого доступа.

### UIManager (`src/ui/UIManager.*`)
Модуль визуального интерфейса. Поднимает дисплей через HAL, привязывает LVGL
к панели (буферы + `flush_cb`), регистрирует экраны и переключает их.

## Поток управления при старте

```
setup()
  → Core::begin()
      → Logger::begin()
      → registerModules()        // добавляем UIManager
      → ModuleManager::initAll()  // UIManager::init(): Display + LVGL + экраны
      → ModuleManager::startAll() // UIManager::start(): показать BootScreen
      → publish(SYS_BOOT_DONE)
loop()
  → Core::loop() → updateAll()
      → UIManager::update(): lv_timer_handler() + active->onUpdate()
          → BootScreen по таймауту → setScreen("Home")
```

## Соглашения

- **Имена модулей** (`IModule::name()`) уникальны и служат тегами логов.
- **Пины железа** — только в `src/hal/board_pins.h`, нигде не хардкодятся.
- **Параметры** (тайминги, яркость) — в `include/varsys_config.h`.
- Новые подсистемы кладём в `src/modules/<Name>/`.

## Модуль ввода (этап 2, готово)

`InputModule` (`src/modules/InputModule/`) опрашивает энкодер и кнопку в
`update()` и публикует в `EventBus`:

| Действие              | Событие                |
|-----------------------|------------------------|
| Поворот по часовой    | `INPUT_ENCODER_CW`     |
| Поворот против часовой| `INPUT_ENCODER_CCW`    |
| Короткое нажатие      | `INPUT_BTN_CLICK`      |
| Долгое нажатие        | `INPUT_BTN_LONG`       |

- **Энкодер** — полношаговый табличный декодер (Ben Buxton): ровно одно
  событие на детент, устойчив к дребезгу, никогда не считает «ложные» шаги.
- **Кнопка** — антидребезг + разделение короткого/долгого нажатия.
- Опрос поллингом с периодом `VARSYS_TICK_MS` (≈200 Гц) — без прерываний,
  достаточно для ручного управления. При очень быстром вращении возможна
  потеря шага, но не ложный отсчёт.

`InputModule` регистрируется **до** `UIManager`, поэтому события доходят до
активного экрана в том же тике. `UIManager` пересылает их в
`Screen::onEvent()`; `HomeScreen` уже обрабатывает навигацию по меню.

Параметры — в `include/varsys_config.h` (`VARSYS_ENCODER_REVERSED`,
`VARSYS_BTN_DEBOUNCE_MS`, `VARSYS_BTN_LONGPRESS_MS`, `VARSYS_BTN_ACTIVE_LOW`).

## Что дальше (этап 3)

Модуль радио `RadioModule` (CC1101): инициализация SPI-шины (общей с дисплеем),
приём/передача sub-GHz, публикация принятых пакетов в `EventBus`.
